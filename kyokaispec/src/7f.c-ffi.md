## The C Interface {#c-ffi}

This section describes Austral's support for calling and exporting C code.

### Importing C Functions {#c-ffi-functions}

To call a foreign C function, declare an ordinary Austral function and define it
with a `Foreign_Import` pragma as its body.

For example, given the C function:

```c
double sum(double* array, size_t length);
```

an Austral declaration can be written as:

```austral
function Sum(array: Pointer[Float64], length: Index): Float64;
```

and defined as:

```austral
pragma Unsafe_Module;

import Austral.Memory (Pointer);

module body Example is
    pragma Foreign_Import(External_Name => "sum");
    function Sum(array: Pointer[Float64], length: Index): Float64 is
    end;
end module body.
```

Foreign imports are only permitted in unsafe modules.

Imported functions may not be generic.

### Exporting Austral Functions {#c-ffi-exports}

Austral can also export functions through a generated C wrapper using the
`Foreign_Export` pragma.

Exported functions:

- cannot be generic
- must use only the subset of types accepted by the export wrapper generator
- are exported with the given external symbol name

### Imported Type Mapping {#c-ffi-import-mapping}

For imported C functions, the compiler currently lowers the following surface
types directly:

| Austral type | C representation |
| --- | --- |
| `Bool` | `au_bool_t` / C boolean representation |
| `Nat8` | `au_nat8_t` |
| `Int8` | `au_int8_t` |
| `Nat16` | `au_nat16_t` |
| `Int16` | `au_int16_t` |
| `Nat32` | `au_nat32_t` |
| `Int32` | `au_int32_t` |
| `Nat64` | `au_nat64_t` |
| `Int64` | `au_int64_t` |
| `Index` | `au_index_t` |
| `ByteSize` | `size_t` |
| `Float32` | `float` |
| `Float64` | `double` |
| `Address[T]` | `T*`-compatible pointer |
| `Pointer[T]` | `T*`-compatible pointer |
| `Span[T, R]` | span struct representation |
| `Span![T, R]` | mutable span struct representation |
| `Fn[...]` | function pointer representation |

The exact lowering is backend-defined by the current C backend.

### Exported Type Mapping {#c-ffi-export-mapping}

Exported functions use a stricter type surface. The wrapper generator accepts:

| Austral type | Exported C representation |
| --- | --- |
| `Bool` | `au_bool_t` |
| `Nat8` | `au_nat8_t` |
| `Int8` | `au_int8_t` |
| `Nat16` | `au_nat16_t` |
| `Int16` | `au_int16_t` |
| `Nat32` | `au_nat32_t` |
| `Int32` | `au_int32_t` |
| `Nat64` | `au_nat64_t` |
| `Int64` | `au_int64_t` |
| `Index` | `au_index_t` |
| `ByteSize` | `size_t` |
| `Float32` | `float` |
| `Float64` | `double` |
| `Address[T]` | `T*` |
| `Fn[...]` | function pointer type |

The following are rejected in exported signatures:

- `Unit`
- named user-defined types
- `Region`
- references `&[...]` and `&![...]`
- spans
- type variables
- `Pointer[T]`

### Example: Imported String Output {#c-ffi-example-puts}

The current compiler accepts signatures like:

```austral
pragma Unsafe_Module;

module body Test is
    pragma Foreign_Import(External_Name => "puts");
    function Put_String(str: Span[Nat8, Static]): Int32 is
    end;
end module body.
```

which demonstrates that imported functions can take `Span[Nat8, Static]` values
directly.

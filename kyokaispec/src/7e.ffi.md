# Foreign Interfaces {#ffi}

This section describes Austral's support for communicating with foreign code.

Austral supports foreign imports and foreign exports through pragmas attached to
ordinary function definitions.

Foreign declarations are part of the unsafe boundary. A **`Foreign_Import`**
definition is rejected unless the enclosing module body includes
module-level `pragma Unsafe_Module;` before the `module body` header (see
`Parser.mly` and `TypingPass.augment_decl` in the compiler). Importing
`Austral.Memory` is likewise rejected in safe modules (`ImportResolution.ml`).
**`Foreign_Export`** is processed without that same unsafe-module guard in the
typechecker path shown there; treat exports as still FFI-adjacent and
backend-constrained as described below.

The two relevant pragmas are:

```austral
pragma Foreign_Import(External_Name => "c_symbol");
pragma Foreign_Export(External_Name => "exported_symbol");
```

`Foreign_Import` binds an Austral function definition to an existing foreign
symbol.

`Foreign_Export` causes the compiler to generate a wrapper that exports an
Austral function through the C ABI.

Imported and exported signatures are not identical:

- imported functions may use Austral types that the C backend knows how to lower
  directly
- exported functions are restricted to the smaller set accepted by the export
  wrapper generator
